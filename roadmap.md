# EcoLens 產品 Roadmap

> 日期：2026-07-06  
> 原則：先證明「可信答案有人用、場域願意付費」，再擴城市、語言與功能。

## Roadmap 準則

- 以成果與驗證 gate 排序，不以功能數量衡量進度。
- 第一個城市沒做準之前，不擴第二個城市。
- 安全與來源 gate 不因 pilot 時程縮減。
- 能用 repo 內 JSON、CLI 與人工流程完成，就不先建 CMS、後台或 Agent。
- 每一階段未達 gate，先修正定位或內容，不自動進下一階段。

## Now：MVP 基線

目前已有：

- 台灣／日本、繁中／日文切換
- 圖片上傳、相機與低頻 live mode
- Gemini 受控辨識、多物件與複合材質拆解
- 11 個規則 key 與保守 unknown
- 可選的官方知識 fallback
- Supabase 回饋、SSE、人工審核 CLI
- rate limit、結構化 log、smoke/eval 工具
- 已通過的 unit/script tests，包含大阪規則安全檢查

目前不能對外宣稱：

- 城市級答案已完整適用
- 支援所有垃圾
- 模型或處置規則具有可發布的準確率
- 系統會從回饋自動進化
- 7 個 seed case 足以作安全 release gate

## Phase 1：可信 pilot 基礎（0–6 週）

### 目標

把 MVP 從可展示，提升成可讓一個真實場域試用且不製造假精準的版本。

### 產品工作

- 訪談 5 位場域經營／清潔人員與 10 位目標使用者。
- 第一個城市固定為大阪市；先完成無合作方的家戶垃圾 POC。
- 明確區分國家、自治體與場域規則。
- 每個已回答結果顯示來源、適用範圍與審核日期。
- 新增「有幫助／沒幫助」。
- 將上傳／單次拍攝設為主流程；live mode 不作預設。

### 技術工作

- 將 `sourceIds` 接到 API `evidence` 與 UI。
- 修正 municipality 只改標籤、不一定改規則的假精準。
- 建立最小 site config 與 `?site=` QR context。
- 寫入匿名 outcome events，不保存影像或 GPS。
- golden dataset 擴充至至少 40 張 vision、60 個 policy、30 個 E2E case。
- 建立規則來源與 `reviewedAt` 驗證。

### 商業工作

- 找到 3 個願意放置 QR 的場域。
- 記錄現行問題基線：每週分類詢問、重分時間或住客抱怨。
- 不先談完整 SaaS 方案，只確認問題、導入意願與可能預算來源。

### Exit gate

- 3 個場域同意 pilot。
- 所有安全 release gate 為 0 違規。
- 來源與 scope 在 UI 可見。
- production smoke test 通過。
- 至少 30 次非團隊成員的有效掃描。

## Phase 2：場域 pilot（第 7–12 週）

### 目標

證明使用者能完成處置，且產品能減少場域的說明或重分成本。

### 產品工作

- 3–5 個同城市場域上線專屬 QR。
- 為每個場域建立最常見 30–50 個物品／部件規則。
- 場域結果清楚說明「這裡的垃圾桶／處理方式」。
- 每週整理 unresolved、沒幫助與錯誤回饋。
- 只在實際回饋需要時修正文案與規則。

### 營運工作

- 每週一次人工規則審核與 release。
- 提供合作場域一頁式摘要：掃描量、解決率、常見 unknown、回饋。
- 用 CLI／試算表交付摘要；暫不建 dashboard。

### Exit gate

- 累積 100 次有效掃描。
- 可信處置完成率 ≥ 70%。
- 已回答結果 helpful ≥ 80%。
- 危險物 false-safe、無效引用與 municipality leakage 皆為 0。
- 至少 2 個場域確認產品解決了可描述的營運問題。
- 至少 1 個場域願意討論付費延續。

未達 gate 時，優先判斷是辨識、規則、使用入口或問題本身不重要；不靠增加功能掩蓋。

## Phase 3：付費與可重複交付（第 4–6 個月）

### 目標

證明同一套導入與維護流程可以重複出售。

### 工作

- 測試每場域年費或專案費，不先建立線上付款。
- 將新場域導入整理為可在一天內完成的 checklist。
- 只有人工週報開始耗時時，才建立最小 partner dashboard。
- 只有合作場域明確需要時加入英文。
- 依 pilot 錯題擴充規則與 golden dataset，不追求開放世界覆蓋。
- 比較導入前後的詢問量、重分時間或住客體驗指標。

### Exit gate

- 至少 3 個付費場域，或等值的有期限付費 pilot。
- 新場域可在一天內完成設定與驗收。
- 每場域規則維護低於每週 2 小時。
- 單次可信處置的模型與營運成本可被場域收入覆蓋。
- 至少兩種場域能使用相同核心產品，沒有大量客製程式。

## Phase 4：第二城市與機構合作（第 6–12 個月）

### 目標

從單點場域工具，成為可由觀光或地方機構採用的在地化服務。

### 候選工作

- 依客戶需求選第二城市，不依城市名氣排序。
- 與 DMO、自治體、校園或物業合作擴大規則資料。
- 申請或參與日本環境省、觀光相關示範計畫。
- 兩次以上客戶要求嵌入時，再提供 API／white-label。
- 客戶要求即時資料時，優先串接既有官方／合作資料，不自建垃圾車或垃圾桶資料庫。
- 規則維護量超出 Git／CLI 能力後，再建最小 CMS 與審核權限。

### Exit gate

- 第二城市能沿用相同 schema、測試與導入流程。
- 機構合作能提供可靠資料或通路，而不只是一次性客製。
- 內容更新、客服與模型成本隨場域增加仍可控。

## Later：有證據才做

- 英文、韓文或其他語言：付費客戶或使用數據證明需求後。
- 完整收運日曆／垃圾桶地圖：取得可靠、可更新的官方或合作資料後。
- Native App／PWA 離線：Web 的相機、QR 或留存明顯受限後。
- 共享 rate-limit store：多 instance 濫用或帳單風險出現後。
- 回饋圖片：無圖無法重現錯誤，且完成明確 opt-in、保存期限與刪除機制後。
- on-device model：雲端成本或隱私成為已量測瓶頸後。
- 自助客戶後台：人工導入與週報成為主要營運成本後。

## Not now

- LangGraph／Multi-Agent
- AI 自動修改規則
- 社群、遊戲化、排行榜、點數
- 廣告與消費者付費牆
- 全球城市一次擴張
- 自建基礎視覺模型
- 為 demo 增加 AR bounding boxes
- 與核心處置決策無關的碳排計算

## 停損條件

出現任一條件時，應重新定位，而不是繼續擴功能：

- 訪談 15 個場域後，仍找不到 3 個願意試用者。
- 使用流程修正兩輪後，helpful 仍低於 60%。
- 場域認為問題存在，但沒有可辨識的成本或預算擁有者。
- 每個場域都需要不同程式碼，無法形成共用規則模型。
- 安全 gate 無法穩定維持為 0。

## 接下來兩週

1. 選定一個城市與目標場域類型。
2. 完成 5 位場域方、10 位使用者訪談。
3. 修正規則 scope 與 evidence 顯示。
4. 建立第一個 site config 與 QR。
5. 擴充最危險、最常見的前 10 個 golden cases。
6. 讓第一個 friendly pilot 完成至少 10 次真實掃描。
